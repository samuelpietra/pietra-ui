## How They Work Together

A typical workflow for adding a new component:

1. **`/new-component Combobox`** — scaffolds the full file structure, writes tests and stories, registers the export
2. **`/code-review Combobox`** — reviews the generated code against all project conventions, reports findings
3. Fix any issues from the review
4. **`/document-component Combobox`** — generates the MDX documentation page with API reference and examples

---

## Conventions Enforced

The skills share a common set of reference files that define project standards. These live under `.claude/skills/code-review/references/` and are checked by both `/new-component` (at creation time) and `/code-review` (at review time):

- **`accessibility.md`** — ARIA roles, keyboard navigation, focus management, screen reader support
- **`patterns-and-conventions.md`** — file organization, CSS tokens, naming, type safety, `forwardRef`/`displayName` patterns
- **`code-quality.md`** — performance, readability rules (no nested ternaries, no `else if`, `switch` must have `default`), component hygiene
- **`testing.md`** — `userEvent` over `fireEvent`, `screen` queries over `container`, what to test and what to skip

When a convention is added or updated in these files, both skills pick it up automatically.

## Adding a New Skill

1. Create a directory under `.claude/skills/` with a `SKILL.md` file
2. Define `name`, `description`, and `allowed-tools` in the frontmatter
3. Write the procedure and expected output format
4. Add any reference files the skill needs under a `references/` subdirectory
